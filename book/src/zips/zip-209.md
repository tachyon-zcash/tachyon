# [ZIP-209](https://zips.z.cash/zip-0209)

**Tracking:** [#109](https://github.com/tachyon-zcash/tachyon/issues/109)

**('Update' ZIP, Category 'Consensus')**

## I. Dependencies

- [Tachyon Shielded Protocol (#103)](https://github.com/tachyon-zcash/tachyon/issues/103) defines `valueBalanceTachyon`

## II. Design Considerations

### Background

[ZIP-209](https://zips.z.cash/zip-0209) is a consensus rule that tracks the running balance of each shielded pool across the blockchain. If accepting a block would make any pool's balance go negative, the block is rejected. Currently it covers three pools:

- **Sprout:** sum of all `vpub_old` minus sum of all `vpub_new`
- **Sapling:** negation of sum of all `valueBalanceSapling`
- **Orchard:** negation of sum of all `valueBalanceOrchard` (zero before NU5)

When Orchard was introduced, its pool was [added directly to ZIP-209](https://github.com/zcash/zips/commit) without a separate Update ZIP.

### Tachyon's Context

This is basically the most straightforward of the Update ZIPs. Tachyon introduces a new shielded pool with a `valueBalanceTachyon` field in the bundle. The same turnstile logic applies: if more value flows out of the Tachyon pool than has ever flowed in, something is wrong and the block should be rejected.

### Do we need a separate Update ZIP?

Same precedent as ZIP-317 and ZIP-221. The [Tachyon Shielded Protocol ZIP (#103)](https://github.com/tachyon-zcash/tachyon/issues/103) will include a "Changes to ZIP 209" section, the editors fold it into ZIP-209's text, and ZIP-209 gets an `Updated-By` header once #103 reaches a Released status.

## III. ZIP Draft

Following the pattern of Orchard's addition to ZIP-209, the Tachyon pool definition is specified here and folded into ZIP-209's [Specification](https://zips.z.cash/zip-0209#specification) section.

### Changes to ZIP 209

In the Specification section of ZIP 209, add the following pool balance definition:

> The **Tachyon chain value pool balance** for a given block chain is the negation of the sum of all `valueBalanceTachyon` fields for transactions in the block chain. (Before the Tachyon network upgrade has activated, the Tachyon chain value pool balance is zero.)

Amend the consensus rule to read:

> If any of the 'Sprout chain value pool balance', 'Sapling chain value pool balance', 'Orchard chain value pool balance', or **'Tachyon chain value pool balance'** would become negative in the block chain created as a result of accepting a block, then all nodes MUST reject the block as invalid.

#### Rationale

The Tachyon pool follows the same pattern as Sapling and Orchard: the bundle carries a signed `valueBalanceTachyon` field representing the net value flowing between the Tachyon pool and the transparent value pool. Negating the sum of these fields across the chain gives the total value held in the pool. The turnstile check prevents exploitation of any bug in the shielded protocol from inflating the pool's balance beyond what was deposited.
