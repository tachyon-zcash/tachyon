# [ZIP-317](https://zips.z.cash/zip-0317)

**Tracking:** [#111](https://github.com/tachyon-zcash/tachyon/issues/111)

**('Update' ZIP, Category 'Consensus')**

## I. Dependencies

- [Tachyon Shielded Protocol (#103)](https://github.com/tachyon-zcash/tachyon/issues/103) defines tachyactions and `nActionsTachyon`

## II. Design Considerations

### Background

[ZIP-313](https://zips.z.cash/zip-0313) defined a flat fee per transaction. [ZIP-317](https://zips.z.cash/zip-0317) replaces it with a proportional fee mechanism where you pay a fee according to:

**conventional_fee** = `marginal_fee` (5000 ZAT) * max(`grace_actions`, `logical_actions`)

where `grace_actions` is usually just a minimum of **2** to minimize DoS vectors, and `logical_actions` is in some sense the key abstraction they define here: a protocol-neutral unit to normalize cost across transaction types.

**logical_actions** = sum of all protocol contributions:
- **transparent** = max(ceil(`tx_in_total_size` / 150), ceil(`tx_out_total_size` / 34))
- **sprout** = 2 * `nJoinSplit`
- **sapling** = max(`nSpends`, `nOutputs`)
- **orchard** = `nActions`
- **ZSA issuance** = `nIssueNotes` + 100 * `nAssetCreations`
- **memo** = max(0, `nMemoChunks` - `freeMemoChunks`)

Logical actions basically just reflect the state and verification cost a transaction imposes on the network. Because fees scale with actions, DoS vectors (like sandblasting attacks) become comparatively harder. ZIP-317 was written after [NU5](https://zips.z.cash/zip-0252) (which introduced Orchard), so it already accounts for Orchard's action model.

Also worth pointing out that ZIP-317 prices what validators must do, not the cost to produce the transaction. Like, generating a proof on a slower device versus a faster device has the same on-chain cost. So the aggregator's work (decompressing proofs, doing recursive composition in uncompressed mode, recompressing) isn't factored in. That's a market / protocol design concern outside the scope of ZIP-317.

### Tachyon's Context

So the natural question we need to trace here is what Tachyon-specific changes need to be amended to ZIP-317? Specifically, how should we define `contribution_Tachyon` in the `conventional_actions` method?

Well that depends obviously on Tachyon's transaction structure, like how tachystamps, tachygrams, and the autonome / aggregate / adjunct lifecycle map to logical actions.

Recall, a **tachyaction** is a {`cv`, `rk`, `sig`} triple that lives in the transaction regardless of stamp type. **Tachygrams** and the **proof** are stamp fields, where the stamp sits alongside actions in a stamped bundle and is stripped entirely in an adjunct.

In existing pools, each action is verified independently. Tachyon throws a wrench here since we have aggregation, which changes the cost model. The on-chain footprint differs by stamp type:

**1. Autonome** (stamped, from wallet)
- `nActionsTachyon` actions, each with `cv` (32B) + `rk` (32B) + `sig` (64B) = **128B** per action
- `nTachygrams` tachygrams, each 32B, state cost (duplicate check across 2-epoch window)
- 1 proof (`PROOF_SIZE_COMPRESSED`), one proof verification
- 1 `anchor` (32B)
- 1 `binding_sig` (64B)
- `value_balance` (8B)

**2. Aggregate** (stamped, from aggregator)
- May have its own actions (a **based** aggregate), or zero actions (**innocent**)
- 1 merged proof, same verification cost as an autonome proof
- Union of all covered tachygrams, but these are also in the adjuncts' logical accounting
- The proof replaces **N** individual proofs with **1**

**3. Adjunct** (stripped, by miner)
- `nActionsTachyon` actions, same per-action data as autonome
- 1 `binding_sig` (64B)
- `value_balance` (8B)
- `tachyonAggregateId` (64B), a `wtxid` reference to the covering aggregate
- **No proof, no tachygrams, no anchor**

But there's an obvious tension here that ZIP-317 never had to consider: the same logical work (actions + tachygrams) gets split across two transactions. The adjunct carries the actions, and the aggregate carries the proof covering those actions.

### Candidate approaches

**1. Fee follows actions (stamp-type agnostic)**

*`contribution_Tachyon` = `nActionsTachyon`*

This is analogous to Orchard where an adjunct with 3 actions pays the same as an autonome with 3 actions. In some sense, the aggregate's proof cost is "free", a kind of network optimization that's not something the fee system prices in. The tachygram state cost is implicitly bundled into the per-action fee.

**2. Fee follows actions + tachygram surplus**

*`contribution_Tachyon` = max(`nActionsTachyon`, `nTachygrams`)*

Since spends publish **2** tachygrams (current + next epoch nullifier) but only **1** action, a spend-heavy transaction would pay more. This prices in the state cost of duplicate checking explicitly. But it also penalizes spends relative to outputs. Maybe that's okay, but we'd need to confer against the ZIP if that asymmetry is fine.

**3. Fee differentiates by stamp type**

```
if stamped (autonome or aggregate):
    contribution_Tachyon = nActionsTachyon + nTachygrams + 1  (+1 for proof verification)

if stripped (adjunct):
    contribution_Tachyon = nActionsTachyon
```

Autonomes/aggregates pay for proof + tachygrams + actions. Adjuncts pay only for actions (reduced fee since proof is offloaded). But there's incentive asymmetries here of course, wallets would obviously prefer their transaction to become an adjunct to pay less.

The deeper problem is that `conventional_actions()` is a pure function of a single transaction. The adjunct's missing cost (tachygrams + proof) gets shifted onto the aggregate, so either the aggregate overpays, or the formula needs cross-transaction knowledge which breaks per-transaction fee isolation.

### Do we need a separate Update ZIP?

Looking at precedent, [ZIP 227](https://zips.z.cash/zip-0227) (ZSA issuance) and [ZIP 231](https://zips.z.cash/zip-0231) (memo bundles) both defined their fee contributions in their own specs, and those were then [folded directly into ZIP-317's text](https://github.com/zcash/zips/commits/main/zips/zip-0317.rst). They didn't create separate 2xxx Update ZIPs for the fee changes. So the [Tachyon Shielded Protocol ZIP (#103)](https://github.com/tachyon-zcash/tachyon/issues/103) will define `contribution_Tachyon` as part of its specification with a "Changes to ZIP 317" section, the editors fold it into ZIP-317's text, and ZIP-317 gets an `Updated-By` header once #103 reaches a Released status. No extra ZIP needed. Regardless, we should discuss the design decisions here.

### Zebra Modifications

The [`conventional_actions()`](https://github.com/ZcashFoundation/zebra/blob/main/zebra-chain/src/transaction/unmined/zip317.rs) method needs a `contribution_Tachyon` term added to the logical actions sum. Everything downstream (`conventional_fee`, `unpaid_actions`, `mempool_checks`) operates on its output and would work unchanged.

## III. ZIP Draft

Following the pattern of [ZIP 227](https://zips.z.cash/zip-0227) (ZSA issuance) and [ZIP 231](https://zips.z.cash/zip-0231) (memo bundles), the fee contribution is defined here and folded into ZIP-317's [Fee Calculation](https://zips.z.cash/zip-0317#fee-calculation) section.

### Changes to ZIP 317

In the [Fee Calculation](https://zips.z.cash/zip-0317#fee-calculation) section of ZIP 317, add the following input variable:

| Input | Units | Description |
| ----- | ----- | ----------- |
| `nActionsTachyon` | number | Tachyon actions count |

Add the following term to the `logical_actions` calculation:

```
contribution_Tachyon = nActionsTachyon
```

Such that `logical_actions` becomes:

```
logical_actions = contribution_Transparent + contribution_Sprout
                + contribution_Sapling + contribution_Orchard
                + contribution_Tachyon
                + contribution_ZSAIssuance + contribution_ZSACreation
                + contribution_Memos
```

#### Rationale

Tachyon actions are structurally analogous to Orchard actions. Each tachyaction contains a value commitment `cv` (32 bytes), a randomized verification key `rk` (32 bytes), and a signature `sig` (64 bytes), for a total of 128 bytes per action. Validators verify one RedPallas signature per action, regardless of stamp type.

Tachygrams and the proof are stamp fields, not action fields. The stamp may be stripped during aggregation (producing an adjunct), but the actions remain in the transaction. Pricing actions alone preserves the property that `conventional_actions()` is a pure function of a single transaction's public fields, without requiring cross-transaction knowledge of which aggregate covers which adjuncts.

The aggregator's proof cost (verification of one recursive proof per stamped bundle) is not separately priced. This follows the same rationale as Orchard, where the Halo 2 proof verification cost is implicitly covered by the per-action fee rather than charged as a separate term.
